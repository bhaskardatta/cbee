# Keystore Handoff

**The single most important artifact in this entire engagement.** Losing it is irrecoverable — without it, neither Spurt nor Swaroop can ever update the Android app again. This doc is the procedure for transferring the keystore from Spurt's machines to Swaroop's, securely, with backups.

---

## What the keystore is

`cbee-release.jks` is a binary file that contains a private cryptographic key. Google Play uses this key's signature to verify that every update to the cbee Android app actually came from you (Swaroop). The first time you uploaded to Play Store (or will upload, depending on timing), this key got registered with your app's listing.

**Lose this file → can't update the app.** You'd have to publish a NEW app with a new bundle ID — losing all ratings, reviews, and installed users.

**Have someone else access this file → they can ship malicious updates to all your users.** Treat it like the private key for your bank account.

---

## What gets transferred

| Item                          | Format                                    |
| ----------------------------- | ----------------------------------------- |
| `cbee-release.jks`            | Binary file, ~2 KB                         |
| Keystore password              | String (long, random)                     |
| Key alias                      | String — currently `cbee`                  |
| Key password                   | String (long, random; may equal keystore password) |
| Date of generation             | For records — when this keystore was created |
| SHA-256 fingerprint            | Verification value — see below             |

## Transfer procedure

### Step 1: Spurt prepares the bundle

On Bhaskar's machine, with the keystore in `secrets/cbee-release.jks`:

```bash
cd secrets

# 1. Get the SHA-256 fingerprint (used to verify integrity at the other end)
keytool -list -v -keystore cbee-release.jks -alias cbee | grep "SHA256:"
# Record this output (looks like SHA256: AB:CD:EF:01:23:45...)

# 2. Create a tarball with the keystore + the passwords file
mkdir -p handoff
cp cbee-release.jks handoff/
cat > handoff/PASSWORDS.txt << EOF
CBEE ANDROID KEYSTORE — handoff to Swaroop B S
Generated: <YYYY-MM-DD>
Generated by: Spurt Studios (Bhaskar Datta P)

Keystore file: cbee-release.jks
Key alias: cbee
Keystore password: <password>
Key password: <password>
SHA-256 fingerprint:
<paste the SHA256: line from step 1>

CRITICAL:
- This file unlocks the ability to publish updates to the cbee Android app.
- DO NOT share, email, slack, or commit this file or its passwords.
- Store the file in a secure location (encrypted drive, 1Password, hardware key vault).
- Make two encrypted backups in separate physical locations.
- If you ever lose this, the cbee Android app cannot be updated. You would have to
  publish a new app with a new bundle ID, losing all reviews and installs.
EOF

# 3. Encrypt the tarball with a strong passphrase (separate from keystore passwords!)
tar -czf cbee-keystore-handoff.tar.gz -C handoff .
gpg --symmetric --cipher-algo AES256 --output cbee-keystore-handoff.tar.gz.gpg cbee-keystore-handoff.tar.gz
# You'll be prompted for the encryption passphrase. Use a long random string.

# 4. Verify the encrypted file
gpg --list-only cbee-keystore-handoff.tar.gz.gpg
ls -lh cbee-keystore-handoff.tar.gz.gpg

# 5. Clean up the unencrypted intermediates
rm cbee-keystore-handoff.tar.gz
rm -rf handoff
```

The artifact is `cbee-keystore-handoff.tar.gz.gpg` — encrypted with a passphrase only you know.

### Step 2: Spurt transfers via two channels (out-of-band)

Send the encrypted file to Swaroop via one channel. Send the GPG passphrase via a DIFFERENT channel. Never both in one place.

**Channel A (the file):** WhatsApp, Google Drive (Swaroop's drive, link shared), or email attachment.

**Channel B (the passphrase):** Voice call, in-person, or a different messaging app entirely (Signal, iMessage if you both have iPhones).

This is the "two-channel rule" — even if one channel is compromised, the file is still encrypted.

### Step 3: Swaroop verifies receipt and integrity

On Swaroop's machine:

```bash
# 1. Decrypt with the passphrase Bhaskar gave verbally/separately
gpg --decrypt cbee-keystore-handoff.tar.gz.gpg > cbee-keystore-handoff.tar.gz

# 2. Extract
mkdir cbee-keystore
tar -xzf cbee-keystore-handoff.tar.gz -C cbee-keystore
ls cbee-keystore/
# Should show: cbee-release.jks  PASSWORDS.txt

# 3. Verify the SHA-256 matches what Spurt recorded in PASSWORDS.txt
keytool -list -v -keystore cbee-keystore/cbee-release.jks -alias cbee | grep "SHA256:"
# Compare against the line in PASSWORDS.txt — must match BYTE FOR BYTE
```

If the SHA-256 matches: the keystore is intact. If it doesn't: STOP, contact Bhaskar — file integrity is broken.

### Step 4: Swaroop establishes long-term storage

Don't leave the keystore on your laptop unprotected. Put it in at least three places:

**Primary:** Encrypted vault on your Mac (or main computer)
```bash
# macOS: create encrypted disk image
hdiutil create -encryption -size 10m -fs APFS -volname "cbee-keys" cbee-keys.dmg
# Mount it, drop the keystore + passwords inside, unmount
```

Or use 1Password / Bitwarden's "Secure Note" + "Attachment" features. Drop the keystore as an attachment and the passwords as a secure note.

**Backup 1:** Print the passwords on paper. Put in a locked drawer or safe. Yes, paper. Paper survives ransomware.

**Backup 2:** Another physical device — an external SSD or USB key stored at a different physical location (parent's house, office, etc.). Encrypt the drive with FileVault / BitLocker.

**Test the backups:** Periodically (every 6 months) verify you can still decrypt and read each backup. Bit rot is real.

### Step 5: Spurt destroys its copy

Once Swaroop confirms receipt and successful storage:

```bash
# On Bhaskar's machine, in the secrets/ folder
shred -u cbee-release.jks
shred -u cbee-keystore-handoff.tar.gz.gpg
# On macOS without shred:
rm -P cbee-release.jks
rm -P cbee-keystore-handoff.tar.gz.gpg
```

Also remove the passwords from Bhaskar's password manager.

After this, Spurt no longer has the ability to sign cbee Android releases. Future Spurt work (Phase 3, etc.) would require Swaroop to re-share or to add Spurt as a Play Console "Developer" with managed signing.

## What if the keystore gets compromised

If you suspect someone else has gotten access to the keystore:

1. **Don't panic.** Most "I think I lost it" turns out to be "I forgot where I put it."
2. Audit who could have had access. Was it on a stolen laptop? Was a backup drive lost?
3. If you accept Play App Signing (Google's managed signing offered on first upload), the production signing key is held by Google. Your "upload key" (which is what cbee-release.jks becomes in that scheme) can be **rotated**:
   - Play Console → App integrity → App signing → Request upload key reset
   - Generate a new keystore with `keytool`
   - Submit the new public certificate
   - Google approves within a few days
   - From then on, sign with the new keystore

If you did NOT accept Play App Signing (i.e., you opted out and use your keystore as the production signing key), you cannot rotate. Compromise = need to publish a new app with a new bundle ID. This is one of the reasons we recommend accepting Play App Signing.

---

## What if you LOSE the keystore

There is no recovery procedure if you didn't accept Play App Signing. You'd have to:
- Publish a new app with a new bundle ID (e.g., `app.cbee.in2`)
- Notify users via the old app to migrate (they have to manually install the new one)
- Lose all ratings, reviews, and installation count

This is why backups matter. THREE backups. In different places. Verified periodically.

## iOS equivalent — no keystore, but the same idea

For iOS, there's no `.jks`-equivalent file you hold. Apple's signing model uses:
- **Distribution certificate** in Apple Developer portal (renewable yearly)
- **Provisioning profiles** (auto-generated when you build)
- **Apple Developer Program account** itself (your login credentials)

The "iOS equivalent of losing the keystore" is losing access to your Apple Developer account. Protect with:
- Strong unique password in 1Password / Bitwarden
- Two-factor authentication enabled (via Apple Account Settings)
- Recovery email + phone kept current
- Account ownership documented somewhere (so if the account holder is unreachable, support can verify ownership)

## Quick checklist for Swaroop

Before signing off on the handoff:

- [ ] Received `cbee-keystore-handoff.tar.gz.gpg` from Bhaskar
- [ ] Received GPG passphrase via a different channel (call, SMS, different app)
- [ ] Decrypted successfully
- [ ] SHA-256 fingerprint verified against PASSWORDS.txt
- [ ] Keystore stored in primary location (encrypted disk image or 1Password)
- [ ] Backup 1 created (paper or printed)
- [ ] Backup 2 created (external drive at second physical location)
- [ ] Tested decryption from all storage locations
- [ ] Confirmed to Bhaskar in writing that handoff is complete and storage is verified
- [ ] Bhaskar destroyed his copy and confirmed in writing

After this, the keystore is your responsibility alone.

---

## When to use the keystore again

Each future Android release requires you to:
- Decrypt the keystore from your secure vault
- Use it to sign a new AAB (Spurt does this if you're still working with them; or any contractor with the file + passwords can)
- Upload to Play Console

The keystore itself never changes. It's the persistent identity of your app.
